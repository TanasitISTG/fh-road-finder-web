STITCHER TOOL DESIGN — for Codex to build
Written by Claude, 2026-05-28
=========================================================

GOAL:
Build a map stitcher that takes multiple FH6 map screenshots,
removes the in-game UI, and combines them into one large seamless
map image. The output can then be fed into fh6_road_finder.py
for a cleaner scan.

=========================================================
UI REGIONS TO CROP (based on sample screenshot analysis)
=========================================================

Screenshot resolution: 2560x1080 (ultrawide) or 2600x1088
User may also use 1920x1080 or other resolutions.

TOP UI (crop out):
  - Location: top-right corner only
  - Content: player level, gamertag, credits (CR counter)
  - Height: approximately 50px from top edge
  - Width: only right ~400px (left side is clean map)
  - Recommendation: crop top 60px from entire image to be safe,
    OR mask only the top-right 400x60 region with surrounding
    map colour (smarter but more complex)

BOTTOM UI (crop out):
  - Location: full-width bottom bar
  - Content: keyboard shortcut hints (Enter, Esc, X, PageUp, etc.)
  - Height: approximately 45px from bottom edge
  - Recommendation: crop bottom 55px from entire image

LEFT/RIGHT EDGES:
  - Clean, no UI. Sometimes thin black bars on ultrawide.
  - Detect and crop any solid black border strips.

SIMPLE APPROACH (recommended for v1):
  Just crop top N pixels and bottom N pixels from every screenshot.
  Let the user configure --crop-top and --crop-bottom with sensible
  defaults (60 and 55).

=========================================================
STITCHING APPROACH
=========================================================

All screenshots are taken at the SAME ZOOM LEVEL (user confirmed).
The map is 2D top-down satellite view with no rotation or tilt.
Screenshots overlap — user pans the map between screenshots.

RECOMMENDED METHOD: OpenCV feature-based stitching

Option A — cv2.Stitcher (simplest):
  stitcher = cv2.Stitcher.create(cv2.Stitcher_SCANS)
  status, result = stitcher.stitch(images)

  Use SCANS mode (not PANORAMA) because the map is flat/planar,
  not cylindrical. SCANS mode uses affine transforms which are
  correct for 2D map panning.

Option B — Manual ORB/SIFT matching (more control):
  1. Detect features with ORB or SIFT
  2. Match features between pairs
  3. Compute homography (or affine since no rotation)
  4. Warp and blend

  More work but gives better control over blending seams.

Start with Option A. Fall back to Option B if stitcher fails
(it can fail if images don't overlap enough).

BLENDING:
  Use multi-band blending or simple feathering at seams.
  cv2.Stitcher handles this automatically in Option A.

=========================================================
FILE STRUCTURE
=========================================================

Add to consolidated folder:
  tools/fh6_map_stitcher.py     — main Python script
  Run_FH6_Map_Stitcher.bat      — BAT launcher

Reuse the same .venv (already has pillow, numpy, opencv-python).

INPUT:  consolidated/input/  (same folder as road finder)
OUTPUT: consolidated/output/stitched_map.png

=========================================================
PYTHON CLI OPTIONS
=========================================================

  --input         Input folder with map screenshots
  --output        Output file path (default: output/stitched_map.png)
  --crop-top      Pixels to crop from top (default: 60)
  --crop-bottom   Pixels to crop from bottom (default: 55)
  --crop-left     Pixels to crop from left (default: 0)
  --crop-right    Pixels to crop from right (default: 0)
  --no-crop       Disable all cropping
  --mode          Stitching mode: "auto" or "manual" (default: auto)
  --quality       Output PNG compression 0-9 (default: 3)

=========================================================
BAT LAUNCHER DESIGN
=========================================================

Simple — no menu needed for v1:

  1. Detect Python / use existing .venv
  2. Check for images in input/
  3. Ask: "Crop UI from screenshots? Y/N [Y]"
  4. Run stitcher
  5. Show result file path
  6. Optionally open the stitched image

=========================================================
PROCESSING STEPS
=========================================================

  1. Load all supported images from input/ (PNG, JPEG, BMP)
  2. Sort by filename (user should name them in order if possible)
  3. Crop UI from each image (top + bottom strips)
  4. Auto-detect and crop black borders if present
  5. Run cv2.Stitcher.create(cv2.Stitcher_SCANS).stitch(images)
  6. If stitcher fails, try with fewer images or report error
  7. Save result as output/stitched_map.png
  8. Print dimensions and file size

=========================================================
EDGE CASES TO HANDLE
=========================================================

  - Only 1 image: just crop UI and save (no stitching needed)
  - Stitcher returns ERR_NEED_MORE_IMGS: images don't overlap enough
  - Stitcher returns ERR_HOMOGRAPHY_EST_FAIL: can't find matches
  - Very large output (8+ screenshots = huge image): warn about RAM
  - Mixed resolutions: resize all to match the first image's height
  - JPEG artifacts: warn that PNG screenshots give better stitching

=========================================================
INCREMENTAL STITCHING (IMPORTANT — must be in v1)
=========================================================

The user cannot capture the entire map in one session. They will:
  1. Take a few screenshots, run the stitcher, see the result
  2. Notice which parts of the map are missing
  3. Go back in-game, take more screenshots of the gaps
  4. Re-run the stitcher with the new screenshots added
  5. Repeat until full map coverage

This means the stitcher MUST support an additive workflow:

DESIGN:

  --previous       Path to a previously stitched output image.
                   If provided, the new screenshots are stitched
                   onto the existing image rather than starting
                   from scratch. This saves time and avoids
                   re-processing old screenshots.

  If --previous is not provided, stitch all images from input/.
  If --previous IS provided:
    1. Load the previous stitched image as the "base canvas"
    2. Load only the NEW screenshots from input/
    3. For each new screenshot, find where it overlaps the base
       using feature matching (ORB/SIFT on the overlap region)
    4. Warp and blend each new screenshot onto the base
    5. Save the updated stitched image

  Alternatively (simpler for v1):
    - Just re-stitch everything from input/ each time.
    - The user adds new screenshots to input/ alongside existing ones.
    - This is slower but much simpler to implement.
    - Since the map is not enormous (maybe 20-30 screenshots max),
      full re-stitch should take under 30 seconds.

  RECOMMENDED FOR V1: Full re-stitch from input/ every time.
  Keep --previous as a v2 optimisation.

COVERAGE VISUALISATION:

  After stitching, generate a COVERAGE MAP:
    output/coverage_map.png

  This shows which parts of the stitched canvas came from actual
  screenshots vs which parts are empty (not yet captured).

  How to build it:
    1. During stitching, track which pixels in the output canvas
       were filled by at least one source image (create a binary
       mask: 1 = covered, 0 = not covered)
    2. Render the coverage map:
       - Covered areas: show the stitched map at full brightness
       - Uncovered areas: show a red or orange checkerboard pattern
         (makes gaps very obvious)
    3. Add a text overlay or console message:
       "Coverage: 72% of bounding box filled. Gaps visible in
        coverage_map.png — take more screenshots of the red areas."

  This is the KEY feature that lets the user know where to go next.

BAT LAUNCHER ADDITIONS:

  After stitching completes, print:
    "Coverage map saved to: output/coverage_map.png"
    "Red/checkerboard areas = parts of the map you haven't captured yet."
    "Add more screenshots to input/ and run again to fill gaps."

  Open coverage_map.png automatically after stitching (like report.html).
